# Recruiter Demo Script

Ideal duration: 3-5 minutes.

Use this script for recruiter calls, portfolio videos, Loom recordings, and live walkthroughs. Keep the language grounded: Humanity Guide OS is a governed AI product demo, not an autonomous system.

## 1. Introduction

Open `/`.

Say:

```text
Humanity Guide OS is a governed AI organization demo. It shows how simulated digital chaos can become reviewable proposals while keeping the human owner in control.
```

Point out:

- the first-screen product story
- the governance contract
- current capabilities vs future blueprints
- visible no-execution boundaries

Avoid promising:

- real OS control
- real filesystem execution
- real Gmail integration
- autonomous agents
- AGI or consciousness

## 2. Product Vision

Stay on `/`.

Explain the vision:

- reduce organizational chaos
- create clear proposals
- make risk and approval visible
- keep execution manual and owner-controlled

Use this framing:

```text
The product value is not raw automation. The value is clarity, governance, and a safe progression from analysis to proposal.
```

## 3. Public Demo

Open `/demo`.

Show:

- public multi-agent conversation UI
- agent selection
- safe prompt response
- frontend to API boundary
- fallback-aware behavior

Say:

```text
This is the public proof of interaction. It is intentionally low-risk and does not execute external actions.
```

## 4. Governance Layer

Open `/lab`.

Show:

- owner-gated private lab
- GENIO governance metadata
- proposal lifecycle
- approval and rejection states
- audit metadata
- controlled execution planning preview

Explain governance-first:

```text
GENIO can classify, propose, route, and block. GENIO does not execute actions or bypass owner approval.
```

## 5. Personal Organizer

Open `/personal`.

Show:

- daily owner-facing organization surface
- dashboard status
- chaos, clarity, and focus scores
- organization zones
- smart manual checklist
- visible safety boundaries

Say:

```text
This is the owner experience. It turns analysis into manual next steps instead of performing destructive actions.
```

## 6. Safety Model

Explain the difference:

- Preview: UI or metadata demonstration that helps the owner understand a possible organization plan.
- Blueprint: documented future architecture that is not active runtime capability.
- Runtime real: live execution against files, email, external APIs, workers, or OS resources. This does not exist in the current project.

Core rules:

- `Proposal != Execution`
- `Approve != Execute`
- simulation-first
- governance-first
- owner-controlled
- reversible architecture

## 7. Architecture Summary

Summarize the stack:

- Next.js 15
- React 19
- TypeScript
- App Router
- Node.js backend/domain layer
- Jest and Testing Library
- Vercel-ready frontend

Summarize the architecture:

- `/` explains the product and recruiter story.
- `/demo` shows safe public interaction.
- `/lab` shows governance, approvals, audit metadata, and simulation-only previews.
- `/personal` shows the owner daily organization experience.

## 8. Closing

Close with:

```text
This project demonstrates product engineering judgment: useful AI interaction, typed architecture, visible governance, and disciplined safety boundaries without overstating capabilities.
```

Final recruiter-safe line:

```text
Everything sensitive remains proposal-only, preview-only, or blueprint-only. No real execution is performed.
```
